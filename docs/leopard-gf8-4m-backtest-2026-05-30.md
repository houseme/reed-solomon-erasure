### 4.3 4M 内部一致性

以 32x16_4m 为基准:

| case | 相对吞吐量 | 说明 |
|------|-----------|------|
| 32x16_4m | 100% | 基准 |
| 64x32_4m | 80.5% | parity 翻倍 (16→32), 吞吐下降 ~20% |
| 96x48_4m | 27.6% | 96 parity, 显著 cache 压力 |
| 128x64_4m | 32.8% | 128 parity, 比 96x48 略高 |

**关键发现**: 128x64 反而比 96x48 吞吐更高, 可能因为 128 是 2 的幂次, radix-4 FFT 更高效。

## 五、4M vs 1M 详细对比

### 5.1 吞吐量比率

| case | 4M/1M 比率 | 分析 |
|------|-----------|------|
| 32x16 | 96.9% | 几乎相同, 4M 略低 |
| 64x32 | 99.2% | 相同 |
| 96x48 | 98.6% | 相同 |
| 128x64 | 96.4% | 几乎相同 |

**结论**: 4M 与 1M 吞吐量差异 <4%, 说明:
1. **瓶颈不在 shard 大小** — FFT 计算量与 shard_size 成正比, 但内存访问模式相同
2. **FlatWork 分配已优化** — 64 字节对齐 + views 缓存, 4M 不会触发额外堆分配
3. **cache 行为一致** — 4M shard 仍在 L3 cache 容量内 (M5 Max 有 48MB L3)

### 5.2 耗时对比

| case | 1M 耗时 | 4M 耗时 | 4M/1M 耗时比 |
|------|---------|---------|-------------|
| 32x16 | 0.081 s | 0.336 s | 4.15x |
| 64x32 | 0.207 s | 0.835 s | 4.03x |
| 96x48 | 0.901 s | 3.654 s | 4.06x |
| 128x64 | 0.988 s | 4.099 s | 4.15x |

**结论**: 4M 耗时约为 1M 的 **4.0-4.2 倍**, 与 shard_size 增长倍数 (4x) 高度一致, 证实算法复杂度为 O(shard_size)。

## 七、4M 场景适用性分析

### 7.1 适用场景

| 场景 | 推荐 shard_size | 原因 |
|------|----------------|------|
| 大文件存储 (视频/镜像) | 4M | 单 shard 4MB, 总数据可达 512MB |
| 分布式存储 (Ceph/MinIO) | 1M-4M | 与对象存储 chunk 对齐 |
| 数据备份 | 4M | 减少 shard 数量, 简化管理 |
| 实时流媒体 | 64K-256K | 低延迟优先 |

### 7.2 4M 性能特征

| 指标 | 值 |
|------|---|
| 最大配置 | 128 data + 64 parity = 192 shards |
| 最大数据量 | 512 MB (128 × 4M) |
| 最大编码时间 | ~4.1 秒 (128x64_4m) |
| 最小吞吐量 | ~105 MB/s (96x48_4m) |
| 最大吞吐量 | ~381 MB/s (32x16_4m) |
| 4M vs 1M 差异 | <4% |

## 八、综合结论

### 8.1 4M 关键发现

1. **4M 与 1M 吞吐量差异 <4%** — shard_size 增长 4 倍, 吞吐几乎不变
2. **耗时与 shard_size 成正比** — 4M 耗时约为 1M 的 4.0-4.2 倍
3. **128x64 反超 96x48** — 128 是 2 的幂次, radix-4 FFT 更高效
4. **FlatWork 64B 对齐有效** — 4M 不触发额外堆分配

### 8.2 优化效果确认

| 优化 | 对 4M 的影响 |
|------|-------------|
| P0: slice_xor u64 | 4M 数据量下 u64 块处理效率更高 |
| P3: FlatWork 对齐 | 4M shard × 192 lanes = 768 MB, 64B 对齐确保 SIMD 效率 |
| P4: dit4 分阶段 | 4M 下 bulk 部分占比更大, 分阶段收益更明显 |
| Auto 策略 | 4M (≥64K) 正确选择 direct |

### 8.3 生产建议

```
# 大文件存储 (推荐)
shard_size = 4 * 1024 * 1024  # 4 MB
data_shards = 32               # 128 MB 总数据
parity_shards = 16             # 64 MB parity
# 预期吞吐: ~380 MB/s, 编码时间: ~0.34s

# 超大文件 (高冗余)
shard_size = 4 * 1024 * 1024  # 4 MB
data_shards = 128              # 512 MB 总数据
parity_shards = 64             # 256 MB parity
# 预期吞吐: ~125 MB/s, 编码时间: ~4.1s
```
